😤O Problema: JOINs em Cascata

A pergunta inocente: "Quem mais comprou livros do mesmo autor que João?"

SQL — 6 JOINs necessários

SELECT DISTINCT c2.nome
FROM clientes c1
JOIN pedidos p1       ON p1.cliente_id = c1.id
JOIN itens_pedido ip1 ON ip1.pedido_id = p1.id
JOIN livros l1        ON l1.id = ip1.livro_id
JOIN autores a        ON a.id = l1.autor_id
JOIN livros l2        ON l2.autor_id = a.id
JOIN itens_pedido ip2 ON ip2.livro_id = l2.id
JOIN pedidos p2       ON p2.id = ip2.pedido_id
JOIN clientes c2      ON c2.id = p2.cliente_id
WHERE c1.nome = 'João'
  AND c2.id != c1.id

Cypher — traversal natural

MATCH (c1:Cliente {nome:'João'})
  <-[:FEITO_POR]-(p1:Pedido)
  -[:CONTEM]->(l1:Livro)
  -[:ESCRITO_POR]->(a:Autor)
  <-[:ESCRITO_POR]-(l2:Livro)
  <-[:CONTEM]-(p2:Pedido)
  -[:FEITO_POR]->(c2:Cliente)
WHERE c2 <> c1
RETURN DISTINCT c2.nome
Banco Relacional — como funciona
O planner SQL precisa calcular o produto cartesiano de cada JOIN e depois filtrar. Com 6 JOINs e tabelas grandes, o custo cresce exponencialmente com a profundidade.
Banco de Grafos — como funciona
O Neo4j percorre os ponteiros físicos dos relacionamentos. Cada hop custa O(1) — não importa o tamanho total do banco, só o tamanho do subgrafo percorrido.
⚠️ Regra prática: Se sua query precisa de mais de 3 JOINs entre dados que são naturalmente relacionados, é um sinal forte de que o modelo de grafos pode ser mais adequado.

🤔Quando Faz Sentido Migrar?

✅ Migre para Neo4j quando...

Relacionamentos são o dado
Redes sociais, recomendações, grafos de conhecimento, supply chain, detecção de fraude — o valor está em quem conhece quem.
Hierarquias e caminhos
Árvores organizacionais, dependências de pacotes, rotas geográficas — consultas de caminho mais curto são nativas no Neo4j.
Schema evolui com frequência
Adicionar um tipo de relacionamento novo não requer ALTER TABLE. Apenas crie os relacionamentos novos.
Queries ad-hoc em profundidade
Atravessar 5, 10, 20 saltos com [:TIPO*1..N] é eficiente. Em SQL seria recursão CTE complexa.

❌ Mantenha o Relacional quando...

Relatórios e agregações tabulares
GROUP BY, SUM, COUNT em milhões de linhas — PostgreSQL com índices bem ajustados ainda vence nisso.
Transações ACID complexas
Operações financeiras com rollback, constraints de integridade estritas, auditorias — o modelo relacional foi feito para isso.
Dados são planos (poucas FKs)
Se as tabelas têm pouco relacionamento entre si, o overhead do modelo de grafos não compensa.
Equipe já domina SQL
Migração tem custo de aprendizagem. Só troque quando o benefício técnico justificar.
💡 Abordagem híbrida: Muitas empresas usam PostgreSQL + Neo4j juntos — o relacional para transações e relatórios, o grafo para recomendações e análise de rede. Não precisa ser uma substituição total!

# Conceito Relacional → Equivalente em Grafo Exemplo da Livraria
1 Tabela → Label autores → :Autor
2 Linha → Nó {id:1, nome:'Ana'} → (a:Autor {id:1})
3 Coluna → Propriedade nome VARCHAR(120) → .nome
4 Chave Estrangeira → Relacionamento autor_id INT REFERENCES → -[:ESCRITO_POR]->
5 Tabela de Junção → Rel. com Propriedades itens_pedido(qtd, preco) → -[:CONTEM {qtd, preco}]->

💡 O conceito mais importante: Tabela de Junção

A tabela de junção (junction table) é o padrão mais importante de migrar, pois ela desaparece como tabela e vira um relacionamento com propriedades:

SQL — 3 tabelas envolvidas

-- pedidos
-- itens_pedido (id, pedido_id, livro_id,
--               quantidade, preco_unitario)
-- livros

SELECT p.id, l.titulo, ip.quantidade
FROM pedidos p
JOIN itens_pedido ip ON ip.pedido_id = p.id
JOIN livros l        ON l.id = ip.livro_id

Cypher — relacionamento carrega os dados

-- itens_pedido virou o relacionamento CONTEM
-- quantidade e preco_unitario ficam no rel.

MATCH (p:Pedido)-[ip:CONTEM]->(l:Livro)
RETURN p.id, l.titulo, ip.quantidade

🔵O Grafo da Livraria — Resultado da Migração

Como o banco fica no Neo4j após a migração: 5 labels, 4 tipos de relacionamento

:Autor ←[:ESCRITO_POR]— :Livro —[:PERTENCE_A]→ :Categoria
:Livro ←[:CONTEM {qtd, preco}]— :Pedido —[:FEITO_POR]→ :Cliente
Relacionamento De → Para Propriedades Origem no SQL
ESCRITO_POR Livro → Autor — livros.autor_id FK
PERTENCE_A Livro → Categoria — livros.categoria_id FK
FEITO_POR Pedido → Cliente — pedidos.cliente_id FK
CONTEM Pedido → Livro quantidade, preco_unitario itens_pedido (tabela de junção)
✅ Note: A tabela itens_pedido não gerou nenhum nó novo! Ela se tornou o relacionamento CONTEM, carregando quantidade e preco_unitario como propriedades da aresta.

⚙️Fase 1 — Preparação do Neo4j

Antes de migrar qualquer dado, faça isso: limpar o banco e criar índices.

🔑 Por que criar índices ANTES de migrar?

Na Fase 2 (relacionamentos), o Cypher precisa fazer MATCH (n:Livro {id: X}) para cada relacionamento. Sem índice, isso é um full scan em todos os nós :Livro — O(N) por relacionamento.

Com índice na propriedade id, cada MATCH é O(1) — acesso direto. Para 500 livros e 5.000 relacionamentos, a diferença é de segundos vs. horas.

Neo4j — Limpeza e criação de índices
def preparar_neo4j(uri, user, password):
    driver = GraphDatabase.driver(uri, auth=(user, password))
    with driver.session() as s:
        # Limpar tudo (cuidado em produção!)
        s.run("MATCH (n) DETACH DELETE n")

        # Criar índices nas propriedades que serão usadas no MATCH
        for label, prop in [('Autor','id'), ('Categoria','id'),
                             ('Livro','id'), ('Cliente','id'), ('Pedido','id')]:
            s.run(f"CREATE INDEX {label.lower()}_id "
                  f"IF NOT EXISTS FOR (n:{label}) ON (n.{prop})")
    driver.close()
IF NOT EXISTS
Torna o script idempotente — pode ser executado múltiplas vezes sem erro.

🔵Fase 2 — Migrando Nós com UNWIND

O padrão UNWIND + MERGE — o coração da migração

Padrão genérico para qualquer entidade
def migrar_entidade(pg_rows, label, cypher_template, driver, batch_size=500):
    """
    pg_rows    : lista de dicts lida do PostgreSQL
    label      : nome do label Neo4j (ex: 'Autor')
    cypher_template: query UNWIND com $lote
    """
    with driver.session() as session:
        for i in range(0, len(pg_rows), batch_size):
            lote = pg_rows[i : i + batch_size]
            session.run(cypher_template, lote=lote)
Exemplo — Migrando :Livro do PostgreSQL para Neo4j
# 1. Ler do PostgreSQL
conn = psycopg2.connect(**PG_CONFIG)
cur  = conn.cursor()
cur.execute("SELECT id, titulo, isbn, preco, avaliacao_media FROM livros")
livros = [{'id': r[0], 'titulo': r[1], 'isbn': r[2],
           'preco': float(r[3]), 'avaliacao_media': r[4]}
          for r in cur.fetchall()]

# 2. Criar nós no Neo4j com UNWIND (processa o batch de uma vez)
cypher = """
    UNWIND $lote AS row
    MERGE (l:Livro {id: row.id})
    SET l.titulo = row.titulo,
        l.isbn   = row.isbn,
        l.preco  = row.preco,
        l.avaliacao_media = row.avaliacao_media
"""
with driver.session() as s:
    s.run(cypher, lote=livros)
UNWIND $lote AS row
Desempacota a lista Python e cria/atualiza um nó para cada elemento. Muito mais eficiente do que um MERGE por chamada.
MERGE vs. CREATE
MERGE é idempotente: se o nó já existe, atualiza. CREATE duplicaria em caso de re-execução. Sempre use MERGE na migração.

🔗Fase 3 — Migrando Relacionamentos (FKs)

Convertendo FK → Relacionamento

Para cada FK, lemos os IDs dos dois lados e criamos uma aresta. O índice garante que o MATCH seja rápido.

Padrão: FK simples (livros.autor_id)
# Ler FKs do PostgreSQL
cur.execute("SELECT id, autor_id FROM livros")
rels = [{'l': r[0], 'a': r[1]} for r in cur.fetchall()]

# Criar relacionamentos com MATCH + MERGE
cypher = """
    UNWIND $lote AS row
    MATCH (l:Livro    {id: row.l})   -- usa o índice livro_id
    MATCH (a:Autor    {id: row.a})   -- usa o índice autor_id
    MERGE (l)-[:ESCRITO_POR]->(a)
"""
with driver.session() as s:
    s.run(cypher, lote=rels)
Tabela de junção com propriedades (itens_pedido)
# Ler a tabela de juncao inteira
cur.execute("SELECT pedido_id, livro_id, quantidade, preco_unitario FROM itens_pedido")
itens = [{'p': r[0], 'l': r[1], 'qtd': r[2], 'preco': float(r[3])}
         for r in cur.fetchall()]

# A tabela de juncao DESAPARECE — vira um relacionamento com props
cypher = """
    UNWIND $lote AS row
    MATCH (p:Pedido {id: row.p})
    MATCH (l:Livro  {id: row.l})
    MERGE (p)-[r:CONTEM {livro_id: row.l}]->(l)
    SET r.quantidade      = row.qtd,
        r.preco_unitario  = row.preco
"""
⚠️ Cuidado com o MERGE de relacionamentos: Inclua uma propriedade identificadora no MERGE (ex: CONTEM {livro_id: row.l}) para evitar relacionamentos duplicados caso o script seja re-executado.

✅Validando a Migração

Após migrar, compare as contagens entre os dois bancos para garantir integridade.

Validação cruzada: PostgreSQL ↔ Neo4j
def validar(pg_config, neo4j_uri, user, password):
    conn   = psycopg2.connect(**pg_config)
    cur    = conn.cursor()
    driver = GraphDatabase.driver(neo4j_uri, auth=(user, password))

    checks = [
        # (descricao, sql_count, cypher_count)
        ('Autores',    'SELECT COUNT(*) FROM autores',
                       'MATCH (n:Autor) RETURN count(n) AS c'),
        ('Livros',     'SELECT COUNT(*) FROM livros',
                       'MATCH (n:Livro) RETURN count(n) AS c'),
        ('ESCRITO_POR','SELECT COUNT(*) FROM livros',
                       'MATCH ()-[r:ESCRITO_POR]->() RETURN count(r) AS c'),
        ('CONTEM',     'SELECT COUNT(*) FROM itens_pedido',
                       'MATCH ()-[r:CONTEM]->() RETURN count(r) AS c'),
    ]
    for desc, sql, cypher in checks:
        cur.execute(sql); pg_n = cur.fetchone()[0]
        with driver.session() as s:
            neo_n = s.run(cypher).single()['c']
        status = 'OK' if pg_n == neo_n else 'DIVERGENCIA'
        print(f"  {desc:<15} PG={pg_n:>6,}  Neo4j={neo_n:>6,}  {status}")

⏱️Benchmark: SQL JOINs vs. Cypher Traversal

Query Q1 — Top vendidos por categoria

SQL: 3 JOINs · Cypher: 3 hops

-- SQL: 3 JOINs
SELECT c.nome, l.titulo,
       SUM(ip.quantidade) AS vendas
FROM categorias c
JOIN livros l        ON l.categoria_id = c.id
JOIN itens_pedido ip ON ip.livro_id    = l.id
JOIN pedidos p       ON p.id = ip.pedido_id
WHERE p.status = 'entregue'
GROUP BY c.nome, l.titulo
ORDER BY vendas DESC LIMIT 10
-- Cypher: traversal direto
MATCH (p:Pedido {status:'entregue'})
      -[ip:CONTEM]->(l:Livro)
      -[:PERTENCE_A]->(c:Categoria)
RETURN c.nome, l.titulo,
       SUM(ip.quantidade) AS vendas
ORDER BY vendas DESC LIMIT 10

Query Q2 — Clientes por autor

SQL: 6 JOINs · Cypher: 6 hops

-- SQL: 6 JOINs — fica grande!
SELECT DISTINCT c2.nome
FROM clientes c1
JOIN pedidos p1  ON p1.cliente_id = c1.id
JOIN itens_pedido ip1 ON ip1.pedido_id = p1.id
JOIN livros l1   ON l1.id = ip1.livro_id
JOIN autores a   ON a.id = l1.autor_id
JOIN livros l2   ON l2.autor_id = a.id
-- ... + 2 JOINs para chegar em c2
-- Cypher: mesma lógica, mais legível
MATCH (c1:Cliente {id:1})
  <-[:FEITO_POR]-(p1)-[:CONTEM]
  ->(l1)-[:ESCRITO_POR]->(a)
  <-[:ESCRITO_POR]-(l2)
  <-[:CONTEM]-(p2)-[:FEITO_POR]
  ->(c2:Cliente)
WHERE c2 <> c1
RETURN DISTINCT c2.nome LIMIT 20
📓 Notebook seção 8: Cada query é executada N_BENCH_REPS = 10 vezes para cálculo de média e mínimo. Isso reduz o ruído de cache e JIT.

💡Armadilhas e Boas Práticas

✅ Boas Práticas

Sempre MERGE, nunca CREATE na migração
CREATE duplica se o script rodar duas vezes. MERGE é idempotente e seguro para reexecuções parciais.
Índices antes dos relacionamentos
Sem índice, o MATCH nas Fases 2 e 3 faz full scan. Com índice, é O(1). Para 500 livros, a diferença pode ser de 1.000x.
Batching com UNWIND
Nunca crie um nó por chamada ao driver. Use UNWIND com lotes de 500–2000 itens para minimizar o round-trip bolt.
Valide após cada fase
Compare contagens PG ↔ Neo4j ao final de cada fase. Muito mais fácil de depurar do que descobrir divergência no final.

❌ Anti-Padrões

Migrar tudo de uma vez sem indexar
Migrar nós e relacionamentos sem índice: o MATCH percorre O(N) nós para cada relacionamento. Para 500 livros e 5.000 rels = 2.500.000 scans.
Recriar o modelo relacional no grafo
Migrar a tabela de junção como nó (:ItemPedido) em vez de relacionamento. Perde o benefício do modelo de grafo.
MATCH sem label
// Varre TODOS os nós
MATCH (n {id: 42}) RETURN n

// Usa o índice correto
MATCH (n:Livro {id: 42}) RETURN n
Sem verificação de duplicatas
Se o processo de migração falhar e for reiniciado sem MERGE, cria duplicatas silenciosas.
