📐Modelando o Banco Escolar

🔵 Nós (Labels) da Escola

:Aluno
nome, idade, matricula, ativo
:Professor
nome, especializacao, registro
:Turma
codigo, ano, periodo
:Disciplina
nome, carga_horaria, creditos
:Escola
nome, cidade, fundacao
:Diretora
(label extra em um Professor)

🔗 Relacionamentos

Relacionamento De → Para Propriedades
MATRICULADO_EM Aluno → Turma desde, status
LECIONA Professor → Disciplina desde, carga_semanal
MINISTRADA_EM Disciplina → Turma horario, sala
TIROU_NOTA Aluno → Disciplina nota, semestre, aprovado
AMIGO_DE Aluno ↔ Aluno desde, nivel_amizade
PERTENCE_A Aluno/Professor → Escola ingresso
DIRIGE Professor → Escola desde, mandato

🟡Exercícios Simples — Parte 1

Consultas básicas para aquecer os neurônios antes da Tia Teteca aparecer com a lista de exercícios 😅

1Liste todos os alunos ativos da escola
🧠 Por que: Usamos WHERE para filtrar só os alunos com ativo: true, ignorando o Toco de Vassoura que foi reprovado.
MATCH (a:Aluno)
WHERE a.ativo = true
RETURN a.nome AS aluno, a.idade AS idade, a.matricula AS matricula
ORDER BY a.nome
2Quais professores existem e o que ensinam?
🧠 Por que: Navegamos pelo relacionamento LECIONA para conectar professores às suas disciplinas. Vemos a Tia Teteca dominando a Matemática!
MATCH (p:Professor)-[:LECIONA]->(d:Disciplina)
RETURN p.nome AS professor, p.especializacao AS area, d.nome AS disciplina
ORDER BY p.nome
3Quais alunos estão na turma 3A?
🧠 Por que: Filtramos pelo código da turma no relacionamento MATRICULADO_EM para encontrar só quem está na turma 3A — o famoso Zé da Manga e seus colegas.
MATCH (a:Aluno)-[:MATRICULADO_EM]->(t:Turma {codigo: "3A"})
RETURN a.nome AS aluno, a.idade AS idade, t.periodo AS periodo
ORDER BY a.nome
4Qual é a nota do Zé da Manga em Matemática?
🧠 Por que: Navegamos pelo relacionamento TIROU_NOTA que carrega a propriedade nota. Spoiler: o Zé da Manga foi mal... muito mal 🤦
MATCH (a:Aluno {nome: "Ze da Manga"})-[n:TIROU_NOTA]->(d:Disciplina)
RETURN a.nome AS aluno, d.nome AS disciplina, n.nota AS nota, n.aprovado AS aprovado

🟡Exercícios Simples — Parte 2

5Quem são os amigos do Zé da Manga?
🧠 Por que: O relacionamento AMIGO_DE é bidirecional na prática social, mas no grafo tem direção. Usamos -[:AMIGO_DE]- sem seta para capturar ambas as direções.
MATCH (ze:Aluno {nome: "Ze da Manga"})-[:AMIGO_DE]-(amigo:Aluno)
RETURN ze.nome AS eu, amigo.nome AS amigo
ORDER BY amigo.nome
6Quem são os alunos aprovados em Matemática?
🧠 Por que: Filtramos o relacionamento pelo aprovado: true. A Pretinha do Norte com 9.8 está no topo — e o Zé da Manga nem aparece 😅
MATCH (a:Aluno)-[n:TIROU_NOTA]->(d:Disciplina {nome: "Matematica"})
WHERE n.aprovado = true
RETURN a.nome AS aluno, n.nota AS nota
ORDER BY n.nota DESC
7Qual professor tem mais tempo de casa na escola?
🧠 Por que: Usamos o ORDER BY na propriedade desde do relacionamento LECIONA para ordenar por antiguidade. Nego Bom, desde 2008, é o decano!
MATCH (p:Professor)-[l:LECIONA]->(d:Disciplina)
RETURN p.nome AS professor, d.nome AS disciplina, l.desde AS ano_inicio
ORDER BY l.desde ASC
8Quantos alunos há em cada turma?
🧠 Por que: Aqui entra a agregação com count(). O Neo4j percorre o grafo e conta quantos alunos chegam em cada turma pelo relacionamento MATRICULADO_EM.
MATCH (a:Aluno)-[:MATRICULADO_EM]->(t:Turma)
RETURN t.codigo AS turma, t.periodo AS periodo, count(a) AS total_alunos
ORDER BY total_alunos DESC

🔴Exercícios Avançados — Parte 1

Agora a Tia Teteca vai aparecer pessoalmente nos exercícios 👩‍🏫💪

A1Encontre alunos que são amigos E estão na mesma turma
🧠 Por que (complexidade): Isso é um padrão duplo no grafo — o Neo4j precisa checar dois relacionamentos ao mesmo tempo: AMIGO_DE e MATRICULADO_EM. Em SQL isso seria um JOIN múltiplo. No grafo é natural!
MATCH (a1:Aluno)-[:AMIGO_DE]-(a2:Aluno)
MATCH (a1)-[:MATRICULADO_EM]->(t:Turma)<-[:MATRICULADO_EM]-(a2)
WHERE id(a1) < id(a2)
RETURN a1.nome AS aluno1, a2.nome AS aluno2, t.codigo AS turma_em_comum
ORDER BY t.codigo
💡 Detalhe: O WHERE id(a1) < id(a2) evita mostrar o par duplicado (Ze-Couve + Couve-Ze) — cada par aparece só uma vez!
A2Qual a média de notas por disciplina? (com alerta de reprovação)
🧠 Por que: Usamos avg() para calcular a média e CASE para criar uma coluna condicional — mostrando "RISCO" quando a média cai abaixo de 6.0. Isso é análise de dados dentro do grafo!
MATCH (a:Aluno)-[n:TIROU_NOTA]->(d:Disciplina)
WITH d.nome AS disciplina,
     avg(n.nota) AS media,
     count(a) AS total_alunos,
     count(CASE WHEN n.aprovado = false THEN 1 END) AS reprovados
RETURN disciplina,
       round(media, 2) AS media_turma,
       total_alunos,
       reprovados,
       CASE WHEN media < 6.0 THEN "⚠️ RISCO" ELSE "✅ OK" END AS status
ORDER BY media ASC

🔴Exercícios Avançados — Parte 2

A3Amigos de amigos — quem o Zé da Manga deveria conhecer?
🧠 Por que: Este é um padrão de recomendação, o mesmo usado por redes sociais como LinkedIn ("pessoas que você pode conhecer"). O Neo4j percorre 2 saltos no grafo de amizade e filtra quem o Zé já conhece.
MATCH (ze:Aluno {nome: "Ze da Manga"})-[:AMIGO_DE]->(amigo:Aluno)-[:AMIGO_DE]->(indicado:Aluno)
WHERE NOT (ze)-[:AMIGO_DE]-(indicado)
  AND ze <> indicado
RETURN indicado.nome AS indicado_para_conhecer,
       collect(amigo.nome) AS via_amigos,
       count(amigo) AS amigos_em_comum
ORDER BY amigos_em_comum DESC
💡 Resultado esperado: O Zé da Manga será conectado à Mariazinha Pé de Pano via Zé da Couve. Fofoca de corredor virou algoritmo de grafo! 😂
A4Caminho mais curto entre dois alunos pela rede de amizades
🧠 Por que: shortestPath() é um algoritmo nativo do Neo4j que encontra o menor número de saltos entre dois nós. Em SQL isso seria impossível sem recursividade complexa. Aqui é uma linha!
MATCH (a:Aluno {nome: "Ze da Manga"}), (b:Aluno {nome: "Pretinha do Norte"})
MATCH path = shortestPath((a)-[:AMIGO_DE*]-(b))
RETURN [no IN nodes(path) | no.nome] AS caminho_de_amizades,
       length(path) AS graus_de_separacao
💡 Lendo o resultado: O caminho mostra por quais amigos a informação viajaria de uma ponta à outra — tipo o "6 graus de separação" de Kevin Bacon, mas na escola do bairro 😄

🔴Exercícios Avançados — Parte 3

A5Qual professor indiretamente influencia mais alunos? (Centralidade)
🧠 Por que: Aqui medimos a influência indireta — o professor que leciona a disciplina onde mais alunos tiraram nota. Isso simula análise de centralidade de grafos, usada em grafos de conhecimento e IA.
MATCH (p:Professor)-[:LECIONA]->(d:Disciplina)<-[:TIROU_NOTA]-(a:Aluno)
WITH p.nome AS professor, d.nome AS disciplina,
     count(a) AS alunos_alcancados,
     avg(a.idade) AS media_idade_turma
RETURN professor, disciplina, alunos_alcancados, round(media_idade_turma, 1) AS media_idade
ORDER BY alunos_alcancados DESC
A6Grafo completo: professor → disciplina → aluno aprovado → amigos dele
🧠 Por que: Este é um padrão de travessia profunda — percorremos 3 relacionamentos diferentes de uma vez. Em bancos relacionais seriam 3 JOINs. No Neo4j é leitura natural do grafo. Isso é o poder do modelo de grafo!
MATCH (p:Professor)-[:LECIONA]->(d:Disciplina)<-[n:TIROU_NOTA]-(a:Aluno)
WHERE n.aprovado = true
OPTIONAL MATCH (a)-[:AMIGO_DE]-(amigo:Aluno)
RETURN p.nome AS professor,
       d.nome AS disciplina,
       a.nome AS aluno_aprovado,
       n.nota AS nota,
       collect(amigo.nome) AS rede_de_amigos
ORDER BY n.nota DESC
💡 OPTIONAL MATCH: Como um LEFT JOIN — retorna o aluno mesmo que ele não tenha amigos cadastrados. O Toco de Vassoura agradece por ainda aparecer na lista 😅
A7Detectar alunos em risco: reprovado E sem amigos que passaram
🧠 Por que: Detecção de padrão negativo — quem está reprovado e não tem amigos aprovados para "puxar" pelo grupo de estudo. Esse tipo de análise é usado em sistemas de apoio estudantil.
MATCH (a:Aluno)-[n:TIROU_NOTA]->(d:Disciplina)
WHERE n.aprovado = false
  AND NOT EXISTS {
    MATCH (a)-[:AMIGO_DE]-(amigo:Aluno)-[:TIROU_NOTA]->(d)
    WHERE amigo IS NOT NULL
  }
RETURN a.nome AS aluno_em_risco, d.nome AS disciplina, n.nota AS nota,
       "Sem amigos aprovados nesta disciplina" AS situacao

⚡Feedback: Performance e Índices

Processos preciosos que fazem o Neo4j voar — a Tia Teteca aprovaria essa eficiência! ⚡

📌 1 — Crie índices para propriedades frequentemente consultadas

Sem índice, o Neo4j precisa varrer todos os nós para encontrar um pelo nome. Com índice, vai direto ao ponto.

// Criar índice na propriedade "nome" de :Aluno
CREATE INDEX aluno_nome IF NOT EXISTS FOR (a:Aluno) ON (a.nome)

// Criar índice na propriedade "nome" de :Professor
CREATE INDEX professor_nome IF NOT EXISTS FOR (p:Professor) ON (p.nome)

// Criar índice na matricula (busca única)
CREATE CONSTRAINT aluno_matricula IF NOT EXISTS
FOR (a:Aluno) REQUIRE a.matricula IS UNIQUE

// Ver todos os índices criados
SHOW INDEXES
🚀 Resultado prático: Uma busca como MATCH (a:Aluno {nome: "Ze da Manga"}) pode ser de 100x a 1000x mais rápida com índice!

📌 2 — Avoid SuperNodes (nós com muitos relacionamentos)

Se um nó como a Escola acumular relacionamentos com 10.000 alunos, consultas nele ficam lentas. Prefira labels + índices a conectar tudo em um só nó.

// ❌ Evite isso quando a escola crescer muito:
MATCH (e:Escola {nome: "Escola Estadual Dona Filo"})<-[:PERTENCE_A]-(a:Aluno)
RETURN count(a)

// ✅ Prefira filtrar diretamente pela propriedade indexada:
MATCH (a:Aluno)
WHERE a.ativo = true
RETURN count(a) AS total_ativos

📌 3 — Use WITH para quebrar queries complexas em etapas

// WITH funciona como um "checkpoint" — você define variáveis intermediárias
// e passa só o necessário para a próxima etapa da query

MATCH (d:Disciplina)<-[n:TIROU_NOTA]-(a:Aluno)
WITH d, avg(n.nota) AS media, count(a) AS total
WHERE media < 6.0
MATCH (p:Professor)-[:LECIONA]->(d)
RETURN p.nome AS professor, d.nome AS disciplina, round(media, 2) AS media, total

🔬Feedback: Diagnose suas Queries

🧪 EXPLAIN — Veja o plano de execução SEM rodar

Como o EXPLAIN do SQL — mostra como o Neo4j planeja executar a query antes de rodar. Útil para identificar gargalos antes de testar em produção.

EXPLAIN
MATCH (a:Aluno {nome: "Ze da Manga"})-[:AMIGO_DE]->(amigo)
RETURN amigo.nome

→ O Neo4j Browser mostra um diagrama de operadores. Procure por NodeIndexSeek (bom!) em vez de AllNodesScan (ruim!).

🔬 PROFILE — Veja o custo real APÓS rodar

PROFILE
MATCH (p:Professor)-[:LECIONA]->(d:Disciplina)<-[n:TIROU_NOTA]-(a:Aluno)
WHERE n.aprovado = true
RETURN p.nome, d.nome, count(a) AS aprovados

→ Mostra db hits (leituras no banco) e rows processadas em cada etapa. Quanto menos db hits, mais eficiente a query!

🛠️ APOC — A caixa de ferramentas do Neo4j

APOC (Awesome Procedures On Cypher) é uma biblioteca de procedimentos que o Neo4j não tem nativamente.

// Importar dados de um JSON externo
CALL apoc.load.json("file:///alunos.json") YIELD value
MERGE (a:Aluno {matricula: value.matricula})
SET a.nome = value.nome, a.idade = value.idade

// Exportar dados do grafo para JSON
CALL apoc.export.json.query(
  "MATCH (a:Aluno) RETURN a",
  "alunos_export.json",
  {}
)

// Algoritmos de centralidade (PageRank dos alunos)
CALL gds.pageRank.stream("alunos-graph")
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).nome AS aluno, score
ORDER BY score DESC

💡Feedback: Padrões Preciosos e Anti-Padrões

✅ Boas Práticas

MERGE antes de CREATE
Sempre use MERGE para evitar nós duplicados ao popular o banco. A Tia Teteca já aparece duas vezes se você usar CREATE sem cuidado! 
MERGE (a:Aluno {matricula: "2024001"})
ON CREATE SET a.nome = "Ze da Manga"
ON MATCH SET a.ultimo_acesso = timestamp()
Sempre filtre com propriedades únicas
Use matricula, registro ou id único — nunca só o nome (pode haver dois "Ze da Manga"!).
DETACH DELETE com cautela
Sempre faça MATCH + RETURN antes de DETACH DELETE. O que foi deletado, não volta. Nem o Toco de Vassoura.

❌ Anti-Padrões

Consulta sem label no MATCH
Isso varre TODOS os nós do banco: 
// ❌ Ruim — sem label
MATCH (n {nome: "Ze da Manga"}) RETURN n

// ✅ Bom — com label
MATCH (a:Aluno {nome: "Ze da Manga"}) RETURN a
Caminhos sem profundidade limitada
Relacionamento [:AMIGO_DE*] sem limite pode travar o banco em grafos grandes. Use [:AMIGO_DE*1..3] para limitar a 3 saltos.
Criar relacionamentos sem checar duplicata
Se rodar o script de amizades duas vezes, o Zé da Manga vai ter 4 amizades com o Zé da Couve. Use MERGE no relacionamento!
