Era Pré-Relacional (1960-1970)

Os Primórdios dos Bancos de Dados

  • Sistemas de Arquivos: Dados armazenados em arquivos sequenciais
  • Modelo Hierárquico: Estrutura em árvore (IBM IMS - 1966)
  • Modelo de Rede: Estrutura em grafo (CODASYL - 1969)
  • Limitações: Navegação complexa, dependência de aplicação, falta de flexibilidade
Exemplo: Sistema bancário com contas organizadas hierarquicamente por agência → cliente → conta

Revolução Relacional (1970-1990)

O Modelo que Mudou Tudo

Edgar F. Codd (1970): Proposta do modelo relacional

  • Características principais: Tabelas, linhas, colunas, chaves primárias/estrangeiras
  • SQL (1974): Linguagem padronizada para consultas
  • Propriedades ACID: Atomicidade, Consistência, Isolamento, Durabilidade
CREATE TABLE usuarios ( id INT PRIMARY KEY, nome VARCHAR(100), email VARCHAR(100) );

Consolidação dos SGBDs Relacionais (1990-2000)

A Era de Ouro do SQL

  • Principais players: Oracle, IBM DB2, Microsoft SQL Server, MySQL, PostgreSQL
  • Padronização: SQL-89, SQL-92, SQL-99
  • Características: Transações robustas, integridade referencial, consultas complexas
  • Domínio absoluto: 95% do mercado de bancos de dados
Aplicações típicas: Sistemas ERP, CRM, financeiros, e-commerce tradicional

O Crescimento da Internet (2000-2005)

Novos Desafios Emergem

  • Explosão de dados: Web 2.0, redes sociais, conteúdo gerado por usuários
  • Escalabilidade: Milhões de usuários simultâneos
  • Variedade de dados: Textos, imagens, vídeos, logs, sensores
  • Velocidade: Necessidade de respostas em tempo real
Exemplos de desafios: Google (bilhões de páginas), Facebook (milhões de usuários), Amazon (catálogo massivo)

Limitações dos Bancos Relacionais

Por que SQL Não Era Mais Suficiente?

  • Escalabilidade Vertical: Limitada pela capacidade de um único servidor
  • Esquema Rígido: Dificuldade para mudanças estruturais
  • Performance: Joins complexos em grandes volumes
  • Distribuição: Complexidade para replicação e sharding
  • Custo: Hardware especializado e licenças caras
Exemplo real: Twitter em 2008 - falhas constantes devido à sobrecarga do MySQL

Teorema CAP (2000)

A Base Teórica do NoSQL

Consistency

Todos os nós veem os mesmos dados simultaneamente

Availability

Sistema permanece operacional

Partition Tolerance

Sistema continua funcionando mesmo com falhas de rede

Teorema: É impossível garantir simultaneamente as três propriedades
Implicação: Bancos relacionais priorizam C+A, NoSQL permite escolher CA, CP ou AP

Nascimento do NoSQL (2005-2010)

A Revolução Silenciosa

Marcos Históricos:

  • Google BigTable (2006): Banco de colunas distribuído
  • Amazon DynamoDB (2007): Banco chave-valor
  • MongoDB (2009): Banco de documentos
  • Termo "NoSQL": Cunhado por Eric Evans em 2009
  • Significado: "Not Only SQL" - não apenas SQL
Marco: Conferência NoSQL em 2009 reuniu principais desenvolvedores

Características Fundamentais do NoSQL

O que Define os Bancos Não Relacionais

  • Escalabilidade Horizontal: Distribuição em múltiplos servidores
  • Esquema Flexível: Estruturas dinâmicas e evolutivas
  • Modelos Diversos: Documento, chave-valor, coluna, grafo
  • Performance: Otimizado para operações específicas
  • BASE: Basically Available, Soft state, Eventually consistent
Filosofia: Sacrificar algumas garantias ACID por performance e escalabilidade

Tipos de Bancos NoSQL

Quatro Categorias Principais

🔑 Chave-Valor

Redis, Amazon DynamoDB, Riak

📄 Documento

MongoDB, CouchDB, Amazon DocumentDB

📊 Coluna

Cassandra, HBase, Google Bigtable

🕸️ Grafo

Neo4j, Amazon Neptune, ArangoDB

Escolha depende: Tipo de dados, padrões de acesso, requisitos de consistência

Bancos Chave-Valor

Simplicidade e Performance

  • Estrutura: Pares chave-valor simples
  • Vantagens: Extremamente rápido, escalável, simples
  • Casos de uso: Cache, sessões, carrinho de compras, contadores
SET usuario:1001 "João Silva" GET usuario:1001 INCR contador_visitas
Empresas: Twitter (cache), GitHub (filas), Pinterest (contadores)

Bancos de Documentos

Flexibilidade Estrutural

  • Estrutura: Documentos JSON/BSON aninhados
  • Vantagens: Esquema flexível, consultas ricas, fácil mapeamento objeto-documento
  • Casos de uso: CMS, catálogos, perfis de usuário, logs
{ "_id": ObjectId("..."), "nome": "João Silva", "enderecos": [ {"tipo": "residencial", "cidade": "São Paulo"}, {"tipo": "comercial", "cidade": "Rio de Janeiro"} ] }

Bancos de Colunas

Otimização para Analytics

  • Estrutura: Dados organizados por famílias de colunas
  • Vantagens: Compressão eficiente, consultas analíticas rápidas, escalabilidade massiva
  • Casos de uso: Data warehousing, analytics, séries temporais
CREATE TABLE vendas ( produto text, ano int, mes int, vendas counter, PRIMARY KEY (produto, ano, mes) );

Bancos de Grafos

Relacionamentos Complexos

  • Estrutura: Nós, arestas e propriedades
  • Vantagens: Consultas de relacionamento eficientes, modelagem natural de redes
  • Casos de uso: Redes sociais, recomendações, detecção de fraudes, roteamento
CREATE (joao:Pessoa {nome: 'João'}) CREATE (maria:Pessoa {nome: 'Maria'}) CREATE (joao)-[:AMIGO_DE]->(maria) MATCH (p:Pessoa)-[:AMIGO_DE]-(amigo) WHERE p.nome = 'João' RETURN amigo.nome

SQL vs NoSQL: Comparação Detalhada

Quando Usar Cada Um?

Aspecto SQL NoSQL
Esquema Fixo, predefinido Flexível, dinâmico
Escalabilidade Vertical (scale-up) Horizontal (scale-out)
Transações ACID completo BASE, eventual consistency
Consultas SQL padronizado APIs específicas
Maturidade Décadas de evolução Tecnologia mais recente
